# 소프트웨어 설계 - AI정리

> 정보처리기사 필기 1과목. 출제 기준(2023.1.1 ~ 2025.12.31) 기준 정리.

---

## 출제 경향

- 대부분 **개념** 위주 출제로, 필요한 점수 확보는 상대적으로 수월한 편.
- **1장(요구사항 확인)과 3장(애플리케이션 설계) 비중이 높음** → 이 두 장 집중 학습.
- 1장: **소프트웨어 생명 주기**, **UML** 집중.
- 3장: **객체지향 구성 요소**, **모듈(결합도·응집도)** 집중.
- 기출문제와 이해를 최우선으로 학습.

## 1. 요구사항 확인

### 1.1 소프트웨어 생명 주기

**구조적(계획 중심, 변경 어려움)**

| 모형 | 설명 |
|------|------|
| **폭포수(Waterfall)** | 보헴 제시. 고전적 생명 주기. 요구사항 변경 어려움. 보완 → 프로토타입 활용 |
| **프로토타입(Prototype)** | 실제 개발될 SW의 견본품. 최종 결과물 예측용 모형 |
| **나선형(Spiral)** | 보헴 제안. 폭포수+프로토타입 장점 + **위험 분석** 추가 |

**애자일(Agile)**  
- 고객 요구사항 변화에 **유연하게 대응**.

**애자일 기반 개발 모형**
- 스크럼(Scrum)
- XP(eXtreme Programming)
- 칸반(Kanban)
- 기능 중심 개발(FDD)

**구조적 개발론 vs 객체지향 개발론** 비교 정리 권장.

---

### 1.2 XP(eXtreme Programming) 기법

- **개요**: 고객 참여, 개발 과정 반복, 개발 생산성 향상.

**XP 핵심 가치**
- 의사소통(Communication)
- 단순성(Simplicity)
- 용기(Courage)
- 존중(Respect)
- 피드백(Feedback)

**XP 주요 실천 방법(영어 함께 암기)**
- **짝 프로그래밍(Pair Programming)**: 두 사람이 한 컴퓨터로 개발
- **공동 코드 소유(Collective Ownership)**: 누구나 코드 수정 가능
- **계속적인 통합(Continuous Integration)**: 작은 단위로 자주 통합
- **리팩토링(Refactoring)**: 코드 구조 개선, 동작은 유지

---

### 1.3 현행 시스템 분석

- **목적**: 개발 범위 확인, 기술 요소·SW·HW·기능·타 시스템과의 정보 교환 파악.
- **파악 절차**: 시스템 구성 → 기능 → 인터페이스 현황 → 아키텍처·SW 구성 → HW·네트워크.
- **주요 분석 항목**: 시스템 기능(계층 분류), 인터페이스(송수신, 연동 데이터·형식, 통신규약), SW 라이선스, HW·네트워크(서버 사양, 이중화, 위치).

---

### 1.4 요구사항 확인

- **요구사항 개발 프로세스**: 도출 → 분석 → 명세 → 확인.
- **분류**: 기능 요구사항(기능성, 완전성, 일관성) / 비기능 요구사항(성능, 보안, 품질, 안정성).
- **도출**: 인터뷰, 설문, 브레인스토밍, 워크숍.
- **확인 기법**: 요구사항 검토, 프로토타이핑, 모델 검증, 인수 테스트.
- **기술적 타당성**: 성능·용량 산정, 상호 운용성, 시장 성숙도·트렌드, 기술적 위험.

---

### 1.5 요구사항 분석·분석 모델

**요구사항 분석 도구**
- **자료흐름도(DFD)**
- **자료사전(DD)**: DFD의 자료를 상세 정의·기록.
- **소단위명세서(Mini-Spec)**

**★DFD 구성 요소**
- **처리(Process)**, **자료 흐름**, **자료 저장소**, **외부 엔티티(단말)**.
- 입력 화살표만 있고 출력이 없어도 됨(필수 대칭 아님).
- 자료저장소 **입력 화살표** = 데이터 입력·수정 의미.

**★자료사전 표기 기호**
- `=`, `+`, `{}`, `[]`, `()`, `**` 등으로 자료 구조·반복·선택 정의 (교재 표 확인).

---

### 1.6 UML(Unified Modeling Language)

- **개요**: 개발자·고객·개발자 간 **의사소통 표준화**. 대표적인 객체지향 모델링 언어.

**구성 요소**
- **사물(Things)**
- **관계(Relationships)**: 1:1, 1:N, N:N 등
- **다이어그램(Diagram)**

**다이어그램 분류**
- **구조적 다이어그램(정적, 6개)**: 클래스, 객체, 컴포넌트, 배치, 복합체, 패키지.
- **행위 다이어그램(동적, 7개)**: 유스케이스, **★순차(Sequence)**, 커뮤니케이션, 상태, 활동, 상호작용 개요, 타이밍.

**★순차(시퀀스) 다이어그램 구성 요소**
- 액터, 객체, 생명선, 실행, 메시지.

**스테레오타입(Stereotype)**
- UML 기본 기능 외 **추가 의미** 표현. `« »`(겔러멧) 안에 표현할 형태 기술.

**★럼바우 분석 기법**
- **객체** → 객체 다이어그램 (객객) / 구조적 다이어그램
- **동적** → 상태 다이어그램 (동상) / 행위 다이어그램
- **기능** → 자료흐름도 DFD (기자)

## 2. 화면 설계

### 2.1 사용자 인터페이스 구분

| 구분 | 설명 |
|------|------|
| **CLI** | Command Line Interface. 텍스트(코드) 형식 |
| **GUI** | Graphical User Interface. 아이콘 등 그래픽 |
| **NUI** | Natural User Interface. 사용자 말·행동(음성, 제스처 등) |

### 2.2 UI 기본 원칙(4가지)

- **직관성**: 누구나 쉽게 이해하고 사용.
- **유효성**: 사용자 목적을 정확하게 달성.
- **학습성**: 누구나 쉽게 배우고 익힘.
- **유연성**: 요구사항을 최대한 수용.

### 2.3 UI 설계 도구

- **목업(Mockup)**: 실제 화면과 유사한 정적 모형.
- **프로토타입(Prototype)**: 테스트 가능한 동적 모형.
- **스토리보드(Storyboard)**: UI 화면·제목·설명 포함 작업 지침서.
- **와이어프레임(Wireframe)**: 레이아웃 협의용 화면 단위 뼈대.

### 2.4 UI 요구사항·설계 프로세스

- UI 시나리오 문서 요건: 추적 용이성, 수정 용이성, 가독성, 일관성, 완전성, 이해성.
- 설계 프로세스: 문제 정의 → 사용자 모델 정의 → 작업 분석 → 오브젝트·기능 정의 → UI 정의 → 디자인 평가.
- 10대 원칙(한국HCI연구회): 가시성, 조작결과 예측성, 일관성, 단순성, 지식배분, 조작오류 방지, 제한사항 선택, 표준화, 행동유도성, 접근성.

## 3. 애플리케이션 설계

### 3.1 소프트웨어 아키텍처·테스트

**Stub(스텁)**
- **하위** 모듈이 아직 없을 때 쓰는 **임시 코드**.
- **상위 → 하위** 순 개발/테스트 시, 하위 모듈 대신 사용.

**Driver(드라이버)**
- **상위** 모듈이 아직 없을 때 쓰는 **임시 코드**.
- **하위 → 상위** 순 개발/테스트 시, 상위 모듈을 호출하는 역할.

---

### 3.2 객체지향(OOP) 구성 요소

| 개념 | 설명 |
|------|------|
| **객체(Object)** | 클래스에서 생성된 **인스턴스** |
| **메시지** | 특정 기능을 수행하도록 요청(함수 호출 등) |
| **클래스** | 공통 속성을 갖는 **객체의 집합** |
| **캡슐화** | 데이터(속성)와 그를 처리하는 함수를 **하나로 묶음** |
| **상속** | 상위 클래스의 특성을 **하위 클래스가 물려받음** |
| **다형성** | 하나의 메시지에 대해 **객체(클래스)별로 다른 방식으로 응답** |

**다형성 구현**
- **오버로딩(Overloading)**: 메소드 이름 같고, **인수 자료형·개수**만 다르게 정의.
- **오버라이딩(Overriding)**: 상위 클래스 메소드와 **이름 같고, 실행 코드만 다르게** 재정의.

---

### 3.3 객체지향 분석·설계

**객체지향 분석 방법론**
- **★E-R 다이어그램** / **Coad와 Yourdon 방법**.

**★객체지향 설계 원칙(SOLID)** (약어 함께 암기)

| 원칙 | 약어 | 내용 |
|------|------|------|
| 단일 책임 | SRP | 객체는 **단 하나의 책임**만 가짐 |
| 개방-폐쇄 | OCP | **기존 코드 변경 없이** 기능 추가 가능하도록 설계 |
| 리스코프 치환 | LSP | 자식 클래스는 부모에서 가능한 행위를 **수행 가능**해야 함 |
| 인터페이스 분리 | ISP | **사용하지 않는 인터페이스**와 의존 관계 맺지 않음 |
| 의존 역전 | DIP | 의존 관계는 **추상성이 높은 클래스**와 맺어야 함 |

---

### 3.4 모듈(결합도·응집도)

**★★★결합도(Coupling)**  
모듈 **간** 상호 의존하는 정도 → **낮을수록 좋음**.

| 순서 | 유형 | 설명 |
|------|------|------|
| 1 | 자료(Data) | 단순 값만 전달 |
| 2 | 스탬프(Stamp) | 배열·구조체 등 전달 |
| 3 | 제어(Control) | 제어 요소(플래그 등) 전달 |
| 4 | 외부(External) | 외부 선언 데이터 참조 |
| 5 | 공통(Common) | 전역 변수 참조·갱신 |
| 6 | 내용(Content) | 다른 모듈 **내부** 직접 참조 (가장 나쁨) |

※ 암기: **내공은 외제를 쓰자** (내용-공통-외부-제어-스탬프-자료)

**★★★응집도(Cohesion)**  
모듈 **내부** 요소들의 관련 정도 → **높을수록 좋음**.

| 순서 | 유형 | 설명 |
|------|------|------|
| 1 | 우연적(Coincidental) | 서로 무관 (가장 낮음) |
| 2 | 논리적(Logical) | 유사한 성격의 처리 |
| 3 | 시간적(Temporal) | 같은 시간에 처리 |
| 4 | 절차적(Procedural) | 순차적으로 실행 |
| 5 | 교환적(Communication) | 같은 입출력으로 다른 기능 |
| 6 | 순차적(Sequential) | 한 출력이 다음 입력 |
| 7 | 기능적(Functional) | 단일 기능 수행 (가장 좋음) |

※ 암기: **귀순병사를 교환하는 절차는 시간적으로 논의해야 함**

---

### 3.5 디자인 패턴

- 객체지향 설계 시 **자주 발생하는 문제**를 피하기 위해 쓰는 **재사용 가능한 설계 패턴**.
- 출제: 1문제 정도. 영상 2회독 등으로 유형 익히기.

**생성**: Abstract Factory, Builder, Factory Method, Prototype, Singleton  
**구조**: Adapter, Bridge, Composite, Decorator, Facade, Flyweight, Proxy  
**행위**: Chain of Responsibility, Command, Interpreter, Iterator, Mediator, Memento, Observer, State, Strategy, Template Method, Visitor

## 4. 인터페이스 설계

### 4.1 인터페이스 요구사항 확인·대상 식별

- 시스템·외부 연동 대상과의 데이터·기능 요구사항 정리.
- 송수신 데이터 항목, 형식, 주기, 예외 조건.
- 인터페이스 표준: 시스템 공통부, 거래 공통부 등.

### 4.2 인터페이스 요구사항 검증

**요구사항 검증 방법**

| 방법 | 성격 | 설명 |
|------|------|------|
| **동료검토** | 비공식 | 명세서 작성자가 직접 설명, 동료가 듣고 결함 발견 |
| **워크스루** | 비공식 | 검토 전 명세서 미리 배포, 사전 검토 후 회의에서 결함 발견 |
| **인스펙션** | 공식 | **작성자 제외**, 전문가들이 명세서 검토하며 결함 발견 |

### 4.3 미들웨어 솔루션 명세

**★미들웨어**  
서로 다른 기종의 HW, 프로토콜, 통신 환경 등을 **연결**하는 소프트웨어.

| 유형 | 역할 |
|------|------|
| **DB** | 데이터베이스와 연결하기 위한 미들웨어 |
| **RPC** | 원격 프로시저를 **로컬 프로시저처럼** 호출 |
| **MOM** | **메시지 기반** 비동기형 메시지 전달 |

### 4.4 인터페이스 상세 설계

- 인터페이스 설계서: 연동 목적, 데이터 명세, 통신 방식, 오류 처리, 보안.
- 통신 방식: 동기/비동기, 배치/실시간, 단방향/양방향.
- 오류 처리: 재시도, 타임아웃, 예외 코드, 로깅·모니터링.

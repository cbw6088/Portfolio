## 1. GoF 디자인 패턴

| 패턴 | 설명 |
|------|------|
| Singleton | 클래스의 인스턴스를 하나만 만들어 공유 |
| Observer | 객체 상태 변화를 관찰자들에게 자동 통지 |
| Adapter | 호환되지 않는 인터페이스를 변환해 함께 동작 |
| Factory Method | 객체 생성을 자식 클래스에 위임 |
| Strategy | 알고리즘을 캡슐화해 상황에 맞게 동적으로 교체 |
| Decorator | 기존 객체를 감싸 기능을 동적으로 추가·확장 |
| Command | 요청을 객체로 캡슐화해 실행·취소·대기열로 처리 |

## 2. 객체지향 5원칙 SOLID

| 원칙 | 설명 |
|------|------|
| LSP (리스코프 치환 원칙) | 자식 타입은 언제나 부모 타입으로 교체 가능 (상위 타입의 규약 위반 금지) |
| OCP (개방-폐쇄 원칙) | 확장에 열림, 수정에 닫힘 |
| DIP (의존성 역전 원칙) | 고수준·저수준 모두 추상화에 의존 |
| SRP (단일 책임 원칙) | 클래스는 하나의 책임·하나의 변경 이유만 가져야 함 |
| ISP (인터페이스 분리 원칙) | 클라이언트가 쓰지 않는 기능에 의존하지 않도록 인터페이스를 잘게 나눔 |

## 4. 트랜잭션의 4대 특성 (ACID)

| 특성 | 설명 |
|------|------|
| 격리성 (Isolation) | 동시 실행 시 "다른 트랜잭션이 끼어들 수 없음" |
| 영속성 (Durability) | 완료된 결과가 "장애가 나도 영구적으로 보존" |
| 원자성 (Atomicity) | 연산이 "모두 반영되거나 전혀 반영되지 않아야" (All or Nothing) |
| 일관성 (Consistency) | 실행 전후 데이터베이스가 "항상 일관된 상태 유지" |

## 5. 암호 알고리즘 / 프로토콜

| 알고리즘·프로토콜 | 설명 |
|-------------------|------|
| RSA | 소인수분해 기반 + 전자서명 + 가장 널리 사용 |
| ECC | 타원곡선 이산로그 기반 + 짧은 키로 동등한 안전성 |
| WEP | 무선랜 초기 보안 프로토콜 + 취약해 TKIP·WPA로 대체 |
| AES·SEED | 대칭키 블록 암호 알고리즘 (암·복호화에 같은 키) |
| MD5 | 해시 함수 (단방향) |
| TKIP | WEP의 취약점을 보완한 후속 무선랜 보안 프로토콜 |
| DES | IBM 개발, 56비트 키, 64비트 블록, 현재 취약, 최초 표준 |
| AES | NIST 개발, 128/192/256비트 키, 128비트 블록, 널리 사용 |

## 7. 사회공학 공격

| 공격 | 설명 |
|------|------|
| 파밍 (Pharming) | DNS·hosts 파일 조작으로 정상 주소를 입력해도 가짜 사이트로 유도 |
| 스미싱 (Smishing) | 문자메시지 (SMS)를 이용 |

| Watering Hole | 표적 집단이 자주 방문하는 사이트에 악성코드를 매복 |

## 9. 네트워크 주소 문제

> 호스트 주소, 서브넷 마스크가 주어짐

| 항목 | 설명 |
|------|------|
| 네트워크 주소 | 호스트 주소와 서브넷 마스크 주소의 AND 연산 |
| 브로드캐스트 주소 | 네트워크 주소에서 서브넷 마스크 주소의 0인 부분을 모두 1로 채움 |
| 배정 불가 주소 | 한 네트워크에서 맨 앞의 네트워크 주소와 맨 뒤의 브로드캐스트 주소는 장비에 배정할 수 없음 |
| 사용 가능한 호스트 수 | 2^(호스트 비트 수) − 2 |

## 10. RAID 기술

| RAID | 설명 |
|------|------|
| 0 | 스트라이핑만 사용해 성능은 가장 높지만, 패리티·미러링이 없어 디스크 하나만 고장 나도 전체 데이터가 손실 |
| 1 | 동일한 데이터를 두 디스크에 복제 (미러링), 공간 효율 50% |
| 5 | 스트라이핑 + 분산 패리티 (최소 3개 디스크)로, 패리티를 이용해 디스크 1개 장애까지 복구 |
| 6 | 스트라이핑 + 이중 패리티 (최소 4개 디스크)로, 디스크 2개 장애까지 복구 |
| 10 | 미러링을 먼저 적용한 뒤 스트라이핑 결합, 최소 4개 디스크 |

## 11. 인터페이스 보안 암호화 프로토콜

| 프로토콜 | 설명 |
|----------|------|
| HTTPS | HTTP에 SSL/TLS 암호화를 결합한 프로토콜로, 웹 통신 채널 전체를 암호화 |
| L2F (Layer 2 Forwarding) | 시스코에서 개발한 데이터 링크 계층 터널링 프로토콜로 UDP를 사용 |

## 12. 구조 패턴

| 패턴 | 설명 |
|------|------|
| Adapter (어댑터) | 서로 호환되지 않는 인터페이스를 중간에서 맞춰 주어, 기존 클래스를 고치지 않고 재사용할 수 있게 하는 구조 패턴 (인터페이스를 변환해 연결하는 것이 목적일 뿐, 객체를 공유해 메모리를 아끼지는 않음) |
| Facade (퍼사드) | 복잡한 서브시스템에 대해 단순한 통합 창구를 제공해, 클라이언트가 내부 구조를 몰라도 기능을 쓸 수 있게 하는 구조 패턴 (사용법을 간단하게 만들 뿐 내부에서 만들어지는 객체 수를 줄여 주지는 않음) |
| Proxy (프록시) | 원래 객체에 바로 접근하지 않고 대리 객체를 거치게 해 접근을 제어하거나 비용을 줄이는 구조 패턴 |
| Decorator (데코레이터) | 원래 클래스는 그대로 둔 채 객체를 감싸서 동적으로 새로운 기능을 덧붙여 확장하는 구조 패턴 |
| Composite (컴포지트) | 객체들을 트리 구조로 구성해 부분-전체 계층을 표현하고, 개별 객체와 복합 객체를 똑같은 방식으로 다루게 하는 구조 패턴 |

## 16. 데이터 링크 계층의 주요 프로토콜

| 프로토콜 | 설명 |
|----------|------|
| HDLC | 비트 중심의 프로토콜로, 점대점 및 다중점 링크를 모두 지원. 데이터 투명성을 보장하기 위해 비트 스터핑 (Bit Stuffing)을 사용하며, 정보 프레임 (I), 감독 프레임 (S), 비번호 프레임 (U)으로 구성 |
| PPP | 점대점 방식의 데이터 링크 프로토콜로, 인터넷 서비스 제공자 (ISP)와 사용자 사이에서 주로 사용. 인증, 압축, 암호화 등 다양한 기능을 제공 |
